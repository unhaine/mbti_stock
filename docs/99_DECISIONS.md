### D-001: React + Vite 선택

**날짜**: 2026-01-15  
**상태**: ✅ 확정

**문제**
- 프론트엔드 프레임워크 선택 필요
- 빠른 개발 환경 필요

**옵션**
1. Create React App (CRA)
2. Next.js
3. Vite + React

**결정**: Vite + React

**이유**
- ✅ **빠른 개발 서버**: HMR이 CRA보다 10배 빠름
- ✅ **간단한 설정**: 복잡한 설정 불필요
- ✅ **번들 크기**: 최적화된 프로덕션 빌드
- ✅ **학습 곡선**: React만 알면 됨 (Next.js보다 단순)
- ✅ **조별과제 적합**: SSR 불필요, 정적 사이트로 충분

**트레이드오프**
- ❌ SSR 없음 (SEO 약간 불리)
- ❌ 파일 기반 라우팅 없음 (React Router 필요)

### D-002: LocalStorage 사용 (Phase 1)

**날짜**: 2026-01-15  
**상태**: ✅ 확정

**문제**
- MVP 단계에서 데이터 저장 방법 필요
- 백엔드 개발 시간 부족

**옵션**
1. LocalStorage
2. IndexedDB
3. Backend API + DB

**결정**: LocalStorage

**이유**
- ✅ **간단함**: 구현 5분
- ✅ **충분한 용량**: 5-10MB (우리 데이터 < 1MB)
- ✅ **서버 불필요**: 배포 비용 0원
- ✅ **빠른 속도**: 동기 API, 즉시 읽기/쓰기
- ✅ **MVP 적합**: 조별과제 범위에 딱 맞음

**트레이드오프**
- ❌ 브라우저 간 동기화 불가
- ❌ 데이터 손실 가능성 (브라우저 캐시 삭제 시)
- ❌ 보안 취약 (민감 정보 저장 불가)

**마이그레이션 계획** (Phase 2)
```
LocalStorage → Backend DB
- 사용자 계정 시스템 추가 시
- 북마크, 포트폴리오 영구 저장 필요 시
```

### D-003: Framer Motion 선택

**날짜**: 2026-01-15  
**상태**: ✅ 확정

**문제**
- 스와이프 제스처 및 애니메이션 구현 필요
- 모바일 앱 같은 UX 제공

**옵션**
1. CSS Transitions
2. React Spring
3. Framer Motion

**결정**: Framer Motion

**이유**
- ✅ **제스처 지원**: drag, swipe 내장
- ✅ **선언적 API**: 코드 가독성 높음
- ✅ **성능**: GPU 가속 사용
- ✅ **문서화**: 예제 풍부
- ✅ **번들 크기**: 약 50KB (gzip 후)

**예시**
```jsx
<motion.div
  drag="x"
  dragConstraints={{ left: 0, right: 0 }}
  onDragEnd={handleSwipe}
>
```

**트레이드오프**
- ❌ 번들 크기 증가 (50KB)
- ❌ 학습 필요 (하지만 문서 좋음)

### D-004: Tailwind CSS 선택

**날짜**: 2026-01-15  
**상태**: ✅ 확정

**문제**
- 빠른 스타일링 필요
- 반응형 디자인 필수

**옵션**
1. Vanilla CSS
2. CSS Modules
3. Styled Components
4. Tailwind CSS

**결정**: Tailwind CSS

**이유**
- ✅ **빠른 개발**: 클래스명만으로 스타일링
- ✅ **일관성**: 디자인 토큰 자동 적용
- ✅ **반응형**: `sm:`, `md:` 등 간편
- ✅ **번들 최적화**: 사용하지 않는 CSS 자동 제거
- ✅ **모바일 우선**: 기본이 모바일 스타일

**예시**
```jsx
<div className="bg-white rounded-2xl p-5 shadow-sm hover:shadow-md transition">
```

**트레이드오프**
- ❌ HTML이 지저분해 보일 수 있음
- ❌ 커스텀 디자인 시 클래스명 길어짐

### D-006: 정적 JSON 데이터 (Phase 1)

**날짜**: 2026-01-15  
**상태**: ✅ 확정

**문제**
- 주식 데이터 제공 방법
- 실시간 데이터 vs 정적 데이터

**옵션**
1. 정적 JSON 파일
2. Yahoo Finance API
3. KRX Open API

**결정**: 정적 JSON 파일

**이유**
- ✅ **간단함**: API 키 불필요
- ✅ **안정성**: API 장애 없음
- ✅ **빠름**: 로컬 파일 읽기
- ✅ **비용**: 0원
- ✅ **MVP 적합**: 엔터테인먼트 목적

**데이터 구조**
```json
{
  "005930": {
    "name": "삼성전자",
    "price": 75000,
    "change": 500,
    "changePercent": 0.67
  }
}
```

**트레이드오프**
- ❌ 실시간 데이터 아님
- ❌ 수동 업데이트 필요

**마이그레이션 계획** (Phase 2)
```
정적 JSON → API 연동
- Yahoo Finance API 또는 KRX API
- 일일 1회 업데이트
```

### D-007: 가상 포트폴리오 (정적 수익률)

**날짜**: 2026-01-15  
**상태**: ✅ 확정

**문제**
- 포트폴리오 수익률 계산 방법
- 실시간 vs 정적

**옵션**
1. 정적 수익률 (고정값)
2. 랜덤 수익률
3. 실시간 계산

**결정**: 정적 수익률 (고정값)

**이유**
- ✅ **간단함**: 계산 불필요
- ✅ **일관성**: 항상 같은 결과
- ✅ **MVP 적합**: 데모용으로 충분

**구현**
```javascript
const portfolio = {
  initialValue: 10000000,
  currentValue: 10250000,  // 고정
  change: 250000,
  changePercent: 2.5
};
```

**트레이드오프**
- ❌ 실제 포트폴리오 아님
- ❌ 재방문 시 변화 없음

**개선 방안** (Phase 2)
```
정적 → 동적
- 일일 랜덤 수익률 (-5% ~ +5%)
- 날짜 기반 시드로 일관성 유지
```

**상태**: ✅ 확정

**문제**
- MBTI 캐릭터 생성 방법
- AI 생성 vs 사전 정의

**옵션**
1. AI 생성 (LLM API)
2. 사전 정의 (16개 고정)

**이유**

- ✅ **안정성**: API 장애 없음
